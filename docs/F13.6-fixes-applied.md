## BUG-05 (CRÍTICO) — Upload sem try-catch

**Arquivo:** `src/app/api/upload/route.ts`

- **Problema:** `compressImage()` chamado duas vezes e `uploadFile()` sem try-catch — qualquer falha derrubava a rota com 500 genérico sem mensagem útil
- **Solução aplicada:**
  - Verificação das 5 variáveis de ambiente R2 no início de cada iteração — retorna 503 se faltar alguma
  - `compressImage()` com try-catch individual — fallback silencioso para o buffer original se a compressão falhar
  - Upload inteiro (`uploadFile`) dentro de try-catch — retorna 502 com mensagem amigável em caso de falha

## BUG-09 (HIGH) — Fail-open no rate limit de posts

**Arquivo:** `src/app/api/posts/route.ts`

- **Problema:** Query de contagem de posts diários usava `count ?? 0` — se a query falhasse, `count` seria `null`, `(null ?? 0) >= limit` era `false`, e o post passava sem verificar o limite
- **Solução:** Adicionada verificação explícita de `countError || count === null` — retorna 503 se a query de contagem falhar (fail-closed)

## BUG-01 (CRÍTICO) — admin/aprovar com dados fake e ações sem efeito

**Arquivo:** `src/app/admin/aprovar/page.tsx`

- **Problema:** `pendingKennels` e `approvedKennels` eram arrays hardcoded; `handleApprove/handleReject` só escreviam no `actionLog` local — nenhuma ação chegava ao banco
- **Solução:** Reescrita completa com Supabase:
  - Fetch real de `kennels WHERE status = 'pending'` com join em `profiles` (owner)
  - Fetch real de `kennels WHERE status = 'approved'`
  - `handleApprove`:
    - `UPDATE kennels SET status='approved', verified_at=NOW(), reviewed_by=user.id`
    - `UPDATE profiles SET role='kennel'` para o dono do canil
  - `handleReject`:
    - `UPDATE kennels SET status='rejected', rejected_reason=reason, reviewed_by=user.id`
  - Toast de feedback para ações bem-sucedidas ou com erro
  - Loading state por ação individual

## BUG-02 (CRÍTICO) — painel/filhotes com CRUD fake

**Arquivo:** `src/app/painel/filhotes/page.tsx`

- **Problema:** `initialPuppies` hardcoded; todo CRUD era React state local — dados sumiam ao recarregar a página; raça era dropdown com apenas "Golden Retriever" e "Beagle"
- **Solução:** Reescrita completa com Supabase:

  - **Delete:** `DELETE FROM puppies WHERE id = X` com confirmação inline
  - Campo "Raça" convertido de `<select>` hardcoded para `<input type="text">` — permite qualquer raça
  - Loading states para operações assíncronas
  - Tratamento de erros com mensagem inline no formulário

## DDL Necessário no Supabase

Execute no **SQL Editor** do Supabase antes de usar as novas funcionalidades:

```sql
-- BUG-01: Campos para approve/reject de canis
ALTER TABLE public.kennels
  ADD COLUMN IF NOT EXISTS rejected_reason TEXT,
  ADD COLUMN IF NOT EXISTS reviewed_by UUID REFERENCES public.profiles(id);

-- BUG-02: Tabela de filhotes (caso ainda não exista)
CREATE TABLE IF NOT EXISTS public.puppies (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  kennel_id    UUID NOT NULL REFERENCES public.kennels(id) ON DELETE CASCADE,
  name         TEXT NOT NULL,
  breed        TEXT NOT NULL,
  sex          TEXT NOT NULL DEFAULT 'male' CHECK (sex IN ('male', 'female')),
  born_at      DATE,
  price        NUMERIC(10, 2),
  status       TEXT NOT NULL DEFAULT 'available'
                 CHECK (status IN ('available', 'reserved', 'sold', 'upcoming')),
  microchipped BOOLEAN NOT NULL DEFAULT FALSE,
  vaccinated   BOOLEAN NOT NULL DEFAULT FALSE,
  description  TEXT,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- RLS para puppies
ALTER TABLE public.puppies ENABLE ROW LEVEL SECURITY;

-- Leitura pública de filhotes disponíveis
CREATE POLICY "puppies_public_read" ON public.puppies
  FOR SELECT USING (status IN ('available', 'upcoming'));

-- Dono do canil pode fazer tudo nos seus filhotes
CREATE POLICY "puppies_kennel_owner_all" ON public.puppies
  FOR ALL
  USING (
    kennel_id IN (
      SELECT id FROM public.kennels WHERE owner_id = auth.uid()
    )
  )
  WITH CHECK (
    kennel_id IN (
      SELECT id FROM public.kennels WHERE owner_id = auth.uid()
    )
  );

-- Trigger para updated_at
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$;

CREATE TRIGGER puppies_updated_at
  BEFORE UPDATE ON public.puppies
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();
```
